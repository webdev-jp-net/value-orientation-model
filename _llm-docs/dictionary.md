# 開発用ネーミング辞書

## 概要
このドキュメントは、開発における統一的なネーミング規約を定めたものです。
コードベース、UI/UX、データベース、API設計における表記揺れを防ぎ、一貫性のある開発を促進します。

## プロジェクト名称

**日本語文脈での呼称**: 価値志向モデル
**英字表記のslug**: value-orientation-model

**用途別指示**:
- 日本語の文脈でプロジェクトを呼称するとき →「価値志向モデル」
- プログラムなど英字表記でプロジェクトを表すslugが必要なとき →「value-orientation-model」

## 基本原則
- **統一性**: 同じ概念には同じ名称を使用
- **可読性**: 開発者が理解しやすい名称を選択
- **国際化対応**: 英語と日本語の対応関係を明確化

## 主要エンティティ

### ユーザー関連
| 日本語   | 英語（コード） | 説明                 |
| -------- | -------------- | -------------------- |
| ユーザー | user           | 一般利用者           |
| 管理者   | admin          | システム管理者       |
| 組織     | group          | アカウントの所属グループ |

## プロダクト関連

### 軸
| 日本語 | 英語（コード） | 説明 |
| -------- | -------------- | -------------------- |
| 帰属志向 | valueLocus | Y軸。価値の所在（内在化・外在化）をあらわした志向 |
| 関係性志向 | boundary | X軸。自己と他者の境界の設け方をあらわした志向 |

### 志向・端点
| 日本語 | 英語（コード） | 説明 |
| -------- | -------------- | -------------------- |
| オーナーシップ | ownership | 帰属志向の上端。責任を主体的に引き受ける選好 |
| コンセンサス | consensus | 帰属志向の下端。責任を場や合意へ移譲する選好 |
| 自立 | diversity | 関係性志向の左端。ダイバーシティ（個の独立性）を選好する志向 |
| 融合 | identityFusion | 関係性志向の右端。一体感（自己と他者の融合）を選好する志向 |

### データ保持・入力
| 日本語 | 英語（コード） | 説明 |
| -------- | -------------- | -------------------- |
| パーソナルプロット | PersonalPlot | 個人ごとの志向データ（1レコード） |
| パーソナルプロットグループ | PersonalPlotGroup | パーソナルプロットの集合体 |
| グループ編集 | GroupEditor | グループ名やプロットリストを編集する機能/コンポーネント |

## フロントエンド命名規則

複数形を表す際は、末尾に `s` をつけず接尾辞として `List` を使用する。

### コンポーネント名
- **形式**: パスカルケース
- **例**: `UserProfile`, `ShopCard`, `ReformRequest`

### CSS クラス名
- **形式**: キャメルケース（CSS Module使用）
- **例**: `header`, `card`, `button`
- **コンディション**: 先頭にハイフン2つ `--isActive`, `--isDisabled`
- **原則**: 可読性重視の意味ベースな短いネーミングを奨励

### ルーティング

**基本原則**: ルートパスの末尾にスラッシュは付けない
